# Agent Rules — checklist obligatorio al tocar UI

> Para cualquier IA (o humano) que escriba o modifique interfaz en este proyecto. Estas reglas son la síntesis operativa de [DESIGN_SYSTEM.md](DESIGN_SYSTEM.md), [COMPONENT_LIBRARY.md](COMPONENT_LIBRARY.md) y [UI_PATTERNS.md](UI_PATTERNS.md) — leé esos tres antes de construir una pantalla nueva; usá esta lista para auto-revisarte antes de dar por terminada la tarea.

## Antes de escribir código

1. **Identificá el tipo de pantalla** (público listado / público detalle / admin listado variante A o B / admin detalle / diálogo / acción rápida mobile-first) y abrí el archivo de referencia más parecido de `UI_PATTERNS.md` §correspondiente. Copiá su estructura, no inventes una nueva salvo que ninguna encaje.
2. **Buscá si el componente que necesitás ya existe** (`COMPONENT_LIBRARY.md`) antes de crear uno — especialmente: loading, empty state, badge de estado, botón de marca, tabla, diálogo de confirmación. Un componente nuevo solo se justifica si de verdad no hay nada parecido.
3. **Buscá si el color de estado que necesitás ya está mapeado** (`*_STATUS_COLORS` en `lib/constants.ts` o el módulo de la entidad) antes de inventar uno inline.

## Reglas no negociables

- **Marca = tokens, no hex (F0, 2026-07-13):** en código nuevo la marca se escribe con los tokens `bg-brand`, `text-brand`, `from-brand via-brand-mid to-brand-2`, `shadow-brand/25`, `<Button variant="brand">` — nunca `#ad45ff` literal. Los hex quedan solo en archivos legacy no migrados (deuda M6); si tocás uno, migrá sus clases de marca de paso.
- **Marca vs. semántica:** el gradiente de marca es para elementos de marca (títulos, CTAs primarios, íconos destacados, barras de acento). Colores Tailwind nombrados (`green`, `red`, `amber`, `blue`, `gray`) son solo para semántica de estado. Nunca uses `violet-600`, `indigo-600`, `purple-500` u otro nombrado como si fuera la marca — es literalmente el bug que se corrigió en `app/admin/partidos/page.tsx` (2026-07-12).
- **Componentes compartidos primero (F0):** hero público → `<PageHero>`; header admin → `<PageHeader>`; KPI admin → `<StatCard>`; badge de estado → `<StatusBadge entity=... status=...>`; estado vacío → `<EmptyState>`; loading con forma conocida → `<SkeletonTable>`/`<SkeletonCards>`; confirmación destructiva → `<ConfirmDialog>`; **tabla del panel → `<DataTable>`**. Todos en `components/shared/` (ver COMPONENT_LIBRARY.md §2b) — no reescribas ese markup a mano.
- **La ficha de jugador es GLOBAL y no tiene dueño (N12, 2026-07-14).** `Player.organizationId` **ya no existe**: hay una ficha por persona en toda la plataforma, identificada por `nationalId` (DNI obligatorio, `@unique` global). Nunca preguntes "¿este jugador es de mi liga?" — no hay campo que responda eso. El permiso y la visibilidad salen de la **participación** (`TeamPlayer → TournamentTeam → Tournament.organizationId`): usá `playerOrgScopeWhere(orgIds)` y `canEditPlayer(user, playerId)` de `lib/playerAuth.ts`, nunca `orgScopeWhere` sobre `Player`. Toda edición se registra en `AuditLog` con `logPlayerChange` (cualquier delegado con el jugador en su plantel puede editar, así que tiene que quedar rastro de quién tocó qué).
- **Un jugador no puede estar en dos equipos del mismo torneo.** El schema **no** lo garantiza: `@@unique([playerId, tournamentTeamId])` solo evita cargarlo dos veces en el *mismo* equipo. La regla se valida en el server (`addPlayerToRoster`). Sí puede estar en varios torneos distintos — esa es justamente la razón de que la ficha sea única y el `TeamPlayer` sea por torneo.
- **Aprobaciones: un solo vocabulario (`ApprovalStatus`) y un solo lugar.** Los flujos donde alguien pide y alguien confirma —delegado de equipo (`TeamManager`), jugador reclamando su ficha (`PlayerClaim`)— comparten el enum `PENDIENTE/APROBADO/RECHAZADO` y la bandeja `/admin/delegados`. Si agregás un tercero, sumalo ahí; no inventes otro enum ni otra pantalla. **Quien aprueba nunca puede ser el solicitante**: verificá que el permiso no dependa del propio pedido pendiente.
- **Un límite se verifica en TODOS los caminos que lo cruzan, y en el momento en que se consume — no solo al crear (auditoría N4b).** El límite no se consume donde el recurso *nace*, sino donde **pasa a ocupar lugar**: aprobar una inscripción, reactivar un torneo archivado, restaurar uno eliminado. Los cuatro bypasses encontrados eran todos de esta forma — un camino nuevo que no preguntaba. Reglas concretas:
  - **La regla de "qué ocupa cupo" se extrae y se comparte** entre el conteo y todos los caminos (`isActiveTournamentStatus`, `effectiveCapacity`). Si solo la sabe el `count()`, el próximo camino la esquiva.
  - **Contá solo lo que ocupa**: un `RECHAZADO` no consume cupo (rechazar no puede costar plata).
  - Antes de dar por hecha una feature que crea/activa algo limitado, **buscá el `assertPlanLimit` correspondiente** y verificá que esté en tu camino.
  - Ante un estado desconocido, **contá** (fallá del lado del negocio): cobrar de más se corrige con un reclamo, regalar el producto no se detecta nunca.
- **No anuncies lo que no existe.** `hasFeature()` es el punto de enforcement de las features de plan; hoy no lo llama nadie porque `exportPdf`/`liveMatch`/`customBranding` no están construidas — pero la landing **sí las anuncia** si se tildan en `/admin/planes`. Al construir una de esas features: implementala, gateala con `hasFeature`, y recién ahí habilitá el switch.
- **Un límite le habla a públicos distintos con mensajes distintos.** El plan de una liga es su relación comercial con la plataforma: al organizador se le muestra con el upsell, **al delegado nunca** (no es su problema y expone a la liga). `effectiveCapacity` devuelve `source: "tournament" | "plan"` justamente para poder decidir eso.
- **Reglas de ventana (cupos, fechas límite) → `lib/`, compartidas por UI y server.** `lib/inscriptions.ts` decide si se puede inscribir y **por qué no**; la UI lo usa para mostrar "quedan 2 cupos / cierra el viernes" y el server para rechazar. Si la UI calcula por su cuenta, termina ofreciendo botones que el server rechaza. Los límites por cantidad se cuentan **solo sobre lo aprobado**, y el control duro va **al aprobar**, no al pedir.
- **Hay DOS ejes de permisos, no uno (N13).** `lib/orgAuth.ts` = "¿gestiona esta liga?" (`OrgRole`: OWNER/ORGANIZADOR/COLABORADOR — personal de la liga). `lib/teamAuth.ts` = "¿es delegado aprobado de este equipo?" (`TeamManager`). **El delegado no es `OrganizationMember` a propósito:** todas las consultas del panel se acotan con `getPanelOrgIds()`, que devuelve *organizaciones*, así que un delegado-miembro vería todos los equipos y jugadores de la liga entera. Si agregás una pantalla para el delegado, va en `/mi-equipo` y se acota con `teamAuth` — nunca en `/admin`.
- **Lógica de negocio pura → `lib/`, con tests (S1).** El generador de fixture (`lib/fixture/*`) y el cálculo de standings (`lib/standings/*`) no tocan Prisma, ni `Date.now()`, ni auth: reciben ids y devuelven un plan. El server action pone permisos, transacción y escritura. Así la parte con reglas de verdad se testea sin base de datos, y los tests encuentran bugs que la UI esconde (el reparto de localía del round-robin tenía un equipo con los 5 partidos de visitante y lo agarró un test, no el navegador). Si un algoritmo nuevo necesita azar, pasale una **semilla**: `Math.random()` hace el test una lotería.
- **Filtrá por el campo que modela el concepto, no por su nombre (S1).** El bracket filtraba `match.phase.name` contra una lista de nombres fijos; cuando A6 borró el modelo `Phase`, el campo dejó de llegar y **todo el display de fases murió en silencio durante meses**. La fase se identifica por `tournamentPhase.type` (`LEAGUE`/`GROUP`/`KNOCKOUT`), que es un enum del schema. Un `name` de texto libre nunca es una condición.
- **Formulario del panel = `<FormSheet>` + los campos de `components/shared/form/fields.tsx` (F3, patrón §11 de UI_PATTERNS).** Nunca un `Dialog` centrado con 20 inputs y scroll interno, nunca un `<Input>` con la cadena de clases escrita a mano, y nunca un formulario sin validación (el `required` del navegador no alcanza: un "número de camiseta: abc" llegaba al server). Las reglas concretas:
  - Validación con **Zod + `zodResolver`**, importando `z` desde **`@/lib/zod-locale`** (locale español global). Escribí un `message` propio solo cuando el genérico no le dice al usuario *cómo* arreglar el campo.
  - El **loading de submit sale de `formState.isSubmitting`** (`onSubmit` async). Prohibido el `useState(isLoading)` que se apaga a mano en un `finally`.
  - Las **fechas viven como string** en el estado del form (lo que hablan los inputs nativos) y se convierten con `lib/date-input.ts`. `toISOString().split("T")[0]` pasa a UTC y **corre el día**.
  - **Estadísticas de standings no se cargan a mano** (PJ/PG/PE/PP/GF/GC/puntos): las calcula el sistema desde los partidos. La API de `tournament-teams` ya no las acepta. Un ajuste manual de puntos va por `bonusPoints`.
- **Navegación del panel = una sola fuente (F3).** Los ítems del menú viven en `lib/constants/admin-nav.ts` (`adminNavItems`): los consumen el sidebar **y** el command palette. Al agregar una ruta admin, sumá el objeto ahí — copiar la lista crea dos menús que divergen en silencio (ya pasó con los dos footers y los dos `STATUS_BADGE`). Toda subpágina admin abre con `<Breadcrumbs>` (patrón §12 de UI_PATTERNS), nunca con un botón "Volver a X", y su último nivel nombra **el recurso** ("Racing"), no el tipo de pantalla ("Detalle del Equipo").
- **Tabla admin = `<DataTable>` (F3).** Nunca una `<Table>` cruda en el panel, y nunca `overflow-x-auto` como "solución" de mobile: el DataTable ya trae orden, búsqueda, filtros, paginación, estado vacío y colapso a cards. Las opciones de un filtro de estado salen **del enum real** (`TOURNAMENT_STATUS_OPTIONS`, `PLAYER_STATUS_OPTIONS`, `MATCH_STATUS`), no de labels escritos a mano: comparar `"En curso"` contra `ACTIVO` era el bug que tenían torneos y jugadores (el filtro no matcheaba nunca y el badge caía siempre al caso `default`).
- **Ningún botón sin handler.** Antes de dar por buena una pantalla, verificá que cada botón haga algo: en el panel había un "Eliminar jugador" sin `onClick` (y sin endpoint DELETE detrás) y un "Programar primer partido" que no abría nada. Un botón que no hace nada es peor que no tener el botón.
- **Eliminar solo lo que no tiene relaciones; el resto se deshabilita** (patrón §8b de UI_PATTERNS, `<DeleteOrDisableButtons>`). Las FK del schema son `onDelete: Cascade`: borrar un jugador o un equipo con historial arrastra sus goles, tarjetas, sanciones y estadísticas de partidos ya jugados. La verificación va **en el server action**, no solo en la UI. Y un registro deshabilitado tiene que salir de **todos** los selectores donde se lo usaría, o no deshabilitaste nada.
- **No declares componentes dentro del render.** Las acciones de fila, celdas, etc. van como función de render (`const renderRowActions = (row) => …`) o como componente a nivel de módulo: un componente creado en cada render remonta su árbol entero (`react-hooks/static-components`).
- **Nada de `setState` síncrono en el cuerpo de un `useEffect`** (`react-hooks/set-state-in-effect`). Los cuatro casos y su solución (los cuatro están aplicados en el repo, 2026-07-14):
  1. **Fetch inicial de una pantalla client** → el fetch va dentro de `startTransition`, y el pendiente de la transición **es** el loading (`const [isLoading, startFetch] = useTransition()`) — no un `useState` que hay que apagar a mano en un `finally`. Referencia: `app/admin/arbitros/page.tsx`, `ManageGoals.tsx`.
  2. **Estado derivado de props** (precargar un form al abrir un diálogo, resetear la página al filtrar) → ajustá **durante el render** comparando una clave con la anterior. Referencia: `app/admin/arbitros/DialogReferee.tsx`, `QuickMatchLoader.tsx`. Y si el valor derivado no hace falta guardarlo, **no lo guardes**: calculalo (era el caso de `tournament-team-form`, que copiaba puntos y diferencia de gol al estado en cada tecla, para nada).
  3. **Leer un sistema externo** (`localStorage`, media queries) → `useSyncExternalStore`. Referencia: `AdminShell.tsx`, `components/theme/theme-provider.tsx`.
  4. **Guard de hidratación** (`mounted`) → `useIsMounted()` (`hooks/use-mounted.ts`), no `useState(false)` + `useEffect(() => setMounted(true))`.
- **Declará las funciones ANTES del `useEffect` que las llama.** Un `const fetchX = ...` debajo del effect que lo usa queda leído en la zona muerta temporal y no se actualiza cuando cambia (`react-hooks/immutability`). Si la función no depende de estado ni props, sacala directamente **fuera** del componente (fue el caso de `deleteImage` en `cloudinary-upload.tsx`).
- **Todo color necesita su par `dark:`.** Si escribís `bg-white`, `text-gray-900`, `border-gray-200`, etc., la línea de al lado (o la misma clase con `dark:`) tiene que existir. Una pantalla sin ninguna clase `dark:` es un bug, no una variante — fue el caso de `GoogleSignUp.tsx` y de todo `admin/arbitros` hasta 2026-07-12.
- **Contraste:** texto de color sobre fondo claro en `600`/`700` como mínimo, nunca `400`/`500` (falla WCAG AA). En dark mode sí se usa `300`/`400` sobre fondos `900/20`-`900/30`.
- **Mobile-first siempre.** Diseñá primero para ~375px de ancho, expandí con `sm:`/`md:`/`lg:`. Objetivos táctiles ≥44×44px en cualquier acción primaria táctil (steppers, botones de acción en cards).
- **Iconos:** solo Lucide (`lucide-react`). Nunca un emoji como ícono funcional/de navegación (sí como acento puntual en copy).
- **Sin datos mock ni endpoints fantasma.** Si una pantalla necesita datos, traelos de la fuente real (API/server action/Prisma) — nunca un array hardcodeado tipo `mockTournaments`, y nunca un filtro de UI que no está conectado a la lógica de filtrado real (verificá que cada `Select`/`Input` de filtro efectivamente se use en el `filter()`/query, no solo que exista visualmente). Fue exactamente el bug de `app/(public)/partidos/page.tsx` hasta 2026-07-12.
- **`AlertDialog` para toda acción destructiva o de alto impacto** (eliminar, suspender, rechazar) — nunca `confirm()`/`alert()` nativos en código nuevo.
- **El copy de una confirmación describe lo que el server hace de verdad (F4).** No la consecuencia más aterradora imaginable: el botón de eliminar torneo decía "no se puede deshacer / se eliminarán todos sus datos" cuando el DELETE es un soft delete que conserva todo. Y si la acción **es** reversible, ofrecé "Deshacer" en el toast (patrón §8c de UI_PATTERNS) — una red de seguridad que existe en el server y no llega a la UI es una red que no existe.
- **Movimiento: una curva y tres duraciones (F4).** `ease-brand` + `duration-fast|base|slow` (DESIGN_SYSTEM §5b). Hover de superficie tocable → `.interactive-surface`, no `hover:-translate-y-1 hover:shadow-xl transition-all duration-300` a mano. Toda animación tiene que expresar causa-efecto; si es decorativa, sobra. **Ojo con `prefers-reduced-motion`:** el CSS global ya frena animaciones y transiciones de CSS, pero **no** un `requestAnimationFrame` — ahí va `usePrefersReducedMotion()`.
- **Superficie de card = `bg-card` (F4).** Ni `bg-white dark:bg-gray-800` a mano: el token ya resuelve los dos temas (era inservible hasta 2026-07-14 porque en dark valía igual que `--background`; ver DESIGN_SYSTEM §7). Al tocar un archivo legacy, migrá sus superficies de paso.
- **Reusar antes que reimplementar.** Antes de escribir una nueva lógica de "goles"/"tarjetas"/"suspensión"/"pagos" en una pantalla nueva, revisá si ya existe un componente que la resuelve (ver el patrón de composición de `QuickMatchLoader.tsx`, que reusa `ManageGoals`/`ManageCards` en vez de reimplementarlos).
- **No dupliques un componente que ya tiene una versión "canónica" y una "legacy".** Si encontrás dos versiones de algo parecido (ej. `components/admin/match-dialog.tsx` vs `DialogAddEditMatch.tsx`), preguntate cuál importa la pantalla en la que estás — no crees una tercera.

## Antes de dar la tarea por terminada

1. `npx tsc --noEmit` limpio.
2. `npx eslint <archivos tocados>` sin errores nuevos (warnings preexistentes de `<img>`/`no-img-element` son aceptables, no los introduzcas de más si podés usar `next/image` sin fricción).
3. `npm run build` en verde.
4. Si es una pantalla nueva o muy modificada: levantar el dev server y verificar con `curl` (o navegador) que la ruta responde 200 y no tiene errores de consola/render. Las rutas `/admin/*` devuelven 401/404 con `curl` sin cookies de Clerk en dev — es esperado, no es un bug (el middleware las protege).
5. Repasá visualmente contra el checklist de accesibilidad del skill `ui-ux-pro-max` (contraste, touch targets, focus visible, reduced-motion) antes de reportar como hecho.
6. **Documentá cualquier hallazgo fuera de alcance en `TODO.md`** (regla ya establecida del proyecto) en vez de arreglarlo silenciosamente si toca una decisión de contenido/negocio (ej. fusionar dos versiones de un footer con contenido real distinto) — arreglalo directo solo si es un fix mecánico sin ambigüedad de contenido.

## Errores ya cometidos en este proyecto — no los repitas

Ver la sección "7. Errores conocidos ya corregidos" de `DESIGN_SYSTEM.md` para el detalle completo. Resumen:
- Clases CSS inventadas/huérfanas que no existen en ningún lado (`bg-golazo-*`).
- Tema oscuro forzado sin soporte real de light mode.
- Paleta ajena a la marca (`violet`/`indigo`/`zinc`/`amber` como color principal).
- Filtros de UI que no filtran nada de verdad.
- Datos mock hardcodeados en una pantalla que ya tiene una API real disponible.
- Dos componentes casi idénticos (ej. dos footers, dos `STATUS_BADGE`) que divergieron en color/contenido sin que nadie lo haya notado — si vas a crear algo que "ya debería existir", buscalo primero.
